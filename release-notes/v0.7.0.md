# HomeOS v0.7.0 — Bootstrap reliability fixes

v0.7.0 hardens the unattended installer and first-boot bootstrap paths using source-level/static validation only. Full ISO and QEMU boot validation remain deferred to the v1.0 orchestrator gate.

## Installer and account setup

- Debian Installer now creates the supported temporary `diadmin` account and the `late_command` robustly renames it to `admin`.
- `/home/admin`, primary group ownership, supplementary groups, sudoers, optional baked SSH keys, and default password expiry are repaired after the rename.
- Public ISOs continue to work without `secrets/authorized_keys`; the default `homeos` password is expired for first login.
- Private ISOs still bake `secrets/authorized_keys` when present, without requiring that file for public builds.
- The optional `/dev/sdb` cache/swap setup now skips mounted media and ISO/UDF devices, and remains non-fatal on single-disk systems.

## First boot and bootstrap

- `homeos-firstboot.service` logs explicit service start/success lines, waits for `/opt/homeos`, and disables itself only after the playbook exits successfully.
- `/var/lib/homeos/bootstrapped` is created by Ansible post-tasks only after the bootstrap roles and success log task complete.
- The firstboot role no longer disables the service from inside the playbook before the bootstrapped marker is written.
- The playbook keeps a bounded 120-second outbound HTTPS network check before roles start.
- Jellyfin `/dev/dri` passthrough is now templated only when the host has `/dev/dri`, so no-GPU systems do not fail Docker stack bootstrap on a missing device.

## CLI and docs drift

- `homeos secure` now repairs SSH key file ownership/modes and validates `sshd -t`, effective public-key auth, and `.ssh/authorized_keys` lookup before disabling password auth and locking the admin password.
- The `homeos` CLI, bash completion, zsh completion, and day-2 docs now agree for secrets, stack, network, backup, portal, Cosmos, and audit commands.
- Troubleshooting docs now cover the reserved `admin` username flow, firstboot marker semantics, network wait behavior, and secure-mode diagnostics.

## Validation

- Shell syntax checks were run for changed shell files.
- YAML parsing was run for changed Ansible YAML files.
- `ansible-playbook --syntax-check bootstrap/install.yml` was run when available.
- Static grep/schema checks were run for preseed account flow, optional SSH key behavior, firstboot marker semantics, and docs/CLI/completion drift.
- No QEMU, VM boot, `make qemu-test`, `make iso`, or equivalent ISO boot/build validation was run by the worker.

## Known limitations

- Real installer execution, hardware disk enumeration, first SSH login, firstboot completion, Docker stack health, and reboot behavior still require orchestrator-owned v1.0 full ISO/QEMU validation.
